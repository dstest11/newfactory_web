# Audit & polish — 2026-05-19

Single-PR audit pass over the homepage SPA + subpages. Five thematic chunks,
each commit minimal and reviewable in isolation. Net effect: tighter UX, fully
keyboard/touch-accessible, smarter scroll model, branded favicon, header that
gets out of the way once the visitor is past the hero.

## Current state (post-audit)

- **Homepage** (`templates/home/index.html.twig`) — Victor 3D paged SPA. Six
  data-page sections (0 hero, 1 STROJE, 2 SLUŽBY, 3 PROČ NÁS, 4 TÝM, 5 KONTAKT).

  `overflow-y: auto`, `max-height: 90 vh`). Page nav driven by Three.js scene

  loads first; project overlay (`assets/styles/app.css`, served via Asset
  Mapper) loads last and wins specificity ties.

## Non-trivial fixes shipped

### 1. Scrolling across page content vs scrolling inside a section

**Symptom.** On the Victor SPA homepage, sections 1–3 have inner overflow
because `max-height: 90 vh` clips long content. When the visitor wheels inside
such a section expecting the inner content to scroll, the page instead
navigated to the next data-page — the inner content was never reachable
without keyboard tricks.

**Root cause.** `public/victor/main.js` registers a single `wheel` /
`touchmove` listener on `window`. Every event triggers `windowWheelOrTouch`,
which after a 1500 ms debounce calls `moveScene()` to TWEEN the Three.js
scene to the next paged section. The handler had **zero awareness** of which
DOM element the event originated from, so wheels over inner-scrollable
sections were treated identically to wheels over the empty hero.

The browser's native scroll behaviour on the `overflow:auto` section was
still running in parallel — but the page jump fired first (or at the same
time) and the inner content never had a chance to be read.

**Fix.** Early-return inside `windowWheelOrTouch`:

```js
const insideScrollable = e.target && typeof e.target.closest === 'function'
    ? e.target.closest('.content-section--scrollable')
    : null
if (insideScrollable) {
    const goingDown = e.touches && e.touches[0]
        ? e.touches[0].pageY < touchStartPosition   // finger moved up = scroll down
        : e.deltaY > 0
    const atTop    = insideScrollable.scrollTop <= 0
    const atBottom = insideScrollable.scrollTop + insideScrollable.clientHeight
                     >= insideScrollable.scrollHeight - 1
    if ((goingDown && !atBottom) || (!goingDown && !atTop)) {
        return
    }
}
// fall through to existing debounce + page-nav logic
```

The decision boundary is the inner content's **edge**, not its centre:

| Where the inner content is              | Wheel down               | Wheel up                 |
|-----------------------------------------|--------------------------|--------------------------|
| Top of inner content (`scrollTop = 0`)  | scroll inner down        | **page-nav up**          |
| Mid-section                             | scroll inner down        | scroll inner up          |
| Bottom (`scrollTop+ch ≥ sh-1`)          | **page-nav down**        | scroll inner up          |

Hits the same boundary semantics most operating systems give a nested
scrollable inside a vertical pager (e.g. iOS native scroll-snap pages with a
scrollable panel). The user can always leave the section by scrolling past
the edge.

**Why `e.target.closest()` over `e.currentTarget` or a custom listener on the
section element.** The wheel listener is on `window`. `currentTarget` is
therefore always `window`. `target` is the deepest element the event was
dispatched on (e.g. an `<li class="nf-machine">` inside the section). Using
`closest('.content-section--scrollable')` walks up the tree from that target
until it finds the scrollable container — or returns null if the wheel
happened over the canvas / hero. That's the cheapest correct check.

**Why we don't `e.preventDefault()`.** When the early-return path triggers,
we want the browser's native overflow scroll to run. Calling `preventDefault`
would cancel that. Not calling it lets the browser scroll the inner section
naturally and the handler simply skips its own page-nav side effect.

**Touch path.** The same early-return covers `touchmove`. `touchStartPosition`
is captured on `touchstart`; on `touchmove` we derive `goingDown` from
`touches[0].pageY < touchStartPosition` (finger moved up = page should
scroll down). Verified at 375 × 667 and 390 × 844 with synthesised touch
events.

**Verified via Chrome MCP.**

```js
// scrollTop = 200 mid-section, inner overflow remaining
s1.scrollTop = 200
inner.dispatchEvent(new WheelEvent('wheel',{deltaY:300,bubbles:true,cancelable:true}))
// after 1.8 s: s1 still visible, s2 still hidden ✓

// scrollTop = scrollHeight - clientHeight (at bottom)
s1.scrollTop = 1363
inner.dispatchEvent(new WheelEvent('wheel',{deltaY:300,bubbles:true,cancelable:true}))
// after 1.8 s: s1 hidden, s2 visible ✓
```

### 2. Section title cleared the fixed header

`.fixed-content-header` is `position: fixed; top: 0; z-index: 1` and visually
~80 – 100 px tall. The override on `nf-section-stroje[data-page="1"]` had
`padding: 1.5 rem`, putting the section title under the header.

Fix: `padding-top: clamp(4.5rem, 12vh, 6.5rem)` on both the base
`.content-section--scrollable` rule and the data-page=1 override. Reserves
72 – 104 px depending on viewport, scaling with `12vh`. Title now lands below
the header at every tested size (320 / 375 / 390 / 1280).

### 3. Header collapses off the hero

The 5 % top padding the Victor template gives `.fixed-content-header` reads as
generous breathing room on the empty hero — but eats vertical space once any
inner section is active. CSS-only fix using `:has()`:

```css
.fixed-content-header {
    padding-top: 10px;                          /* default */
    transition: padding-top 0.4s ease;
}
.ui-wrapper:has(.content-section[data-page="0"]:not(.section--hidden))
    .fixed-content-header {
    padding-top: 5%;                            /* generous on hero only */
}
```

On 1280-wide desktop: 64 px ↔ 10 px. On 375-wide mobile: 18.75 px ↔ 10 px.
Horizontal padding unchanged. Transition synchronises with the 1 s scene
tween. `:has()` is universal in Chrome 105 + / Safari 15.4 + / FF 121 +;
legacy browsers fall back to the compact value which is still readable.

### 4. /victor/main.js cache-bust

`nginx` ships `/victor/*` with `Cache-Control: max-age=432000` (5 days), so
returning visitors would not see #1 / #5 until that window expired. Added
`?v=20260519` to the script tag in `_victor_base.html.twig`. Bump the version
whenever `/victor/main.js` (or `ui.js` / `slider.js`) changes.

### 5. Tap targets on mobile (≥ 44 × 44 px)

Two interactive controls in the fixed header were under WCAG/Apple's 44 × 44
minimum on touch viewports:

- **Burger** — Victor visual is 25 × 13 px (35 × 13 on desktop). Expanded the
  hit surface via `::before { inset: -16px -12px }` to 49 × 45. The visual
  doesn't move; the pseudo-element is absolute-positioned over the parent
  and captures pointer events that bubble up to `.burger`'s click handler.
  Verified via `elementFromPoint` outside the visual.
- **Header KONTAKT CTA** — was 41 px tall (3 px under). Added
  `min-height: 44px; display: inline-flex; align-items: center` — desktop
  already exceeded this, mobile now crosses the threshold.

## Bonus changes (same PR, lower stakes)

- `prefers-reduced-motion: reduce` global rule — kills loader cascade,
  scroll-letter drops and content-section transitions for users with the
  preference.
- Skip-to-content link (`.nf-skip-link`) targeting `#main` on every page.
- Default `:focus-visible` outline for `a / button / [role=button] / summary`
  and for form inputs (was `outline: none` with no replacement).
- Burger / main menu items get proper `role` + `aria-label` + `tabindex` so
  screen reader / keyboard users can operate them.
- Scroll-letter "S/C/R/O/L/L" cluster gets `aria-hidden` per letter and a
  single `aria-label="Scrollujte"` on the container.
- Team photos have name-and-role `alt` (was identical `alt="Team photo"`).
- Decorative SVG icons get `alt="" aria-hidden`.
- KONTAKT anchor in the header was a `<button data-page="5">` — now an
  `<a href="/kontakt">`, works without JS.
- Open Graph + Twitter card meta + `<link rel="canonical">` on every page.
- Favicon: new `public/favicon.svg` — bold orange "N" monogram on dark
  rounded square (#FF6600 on #0a0a0a, brand match). Legacy ICO kept as
  fallback for old crawlers.
- Footer year is now server-rendered (was empty span filled by JS).
- Placeholder phone `+420 777 123 456` and empty `href="#"` social links
  hidden behind `{% if false %}` with TODO-comment guards until real values
  exist.
- `theme-color` aligned with `--color-bg` (#141414 → #0a0a0a).
- `<link rel="icon" type="image/png" href="…favicon.ico">` mime mismatch
  corrected to `image/x-icon` (file is ICO).

## Out of scope

These items came up during the audit but were skipped because they need
design / content decisions, not a quick template tweak:

- **Multi-`<h1>` on the homepage** — five data-page sections each render an
  `<h1>`. Semantic restructuring requires choosing how to expose the IA to
  assistive tech (single page-wide h1 with section h2s, vs. ARIA landmarks
  per section, vs. accept the multi-h1 HTML5 sectioning model).
- **Portfolio images as product hero photos** — `/produkty/{slug}` currently
  pulls `portfolio{1..3}.jpg` from the Victor template as the product
  portrait. Real product photography is a content task.
- **PHPUnit functional test setup** — `framework.test: true` is missing in
  `config/packages/test/`, so 13 of 32 tests error on `WebTestCase::createClient()`.
  Pre-existing, not introduced by this audit.
